Differences between software components and services. The important consequences of these differences

The difference between software components and services can be defined based on the criteria:
By the purpose: Components are pre-formed pieces that are included in other applications and are not supposed to operate 'on their own'. Components help to build system from smaller separate pieces (components) into one system. A service is a self-contained program that can function on its own and can be integrated into more complex systems. The service allows different software to exchange data over the internet. The software that requests data is called a service requester, whereas the software that processes the request and provides the data is called a service provider.
By separation of functionality: Components usually consists of related data and functions. They are more modular and cohesive whereas services can have various functionality and provide several different services at the same time.
By accessibility and interaction with other systems: Components are accessed through “require” and “provide” interfaces, remote procedures or method calls. The web services can be accessed by means of SOAP – “Simple Access Object Protocol”, WSDL – “Web Service Description Language”, UDDI – “Universal Description, Discovery, and Integration” or REST - “Representation State Transfer Architecture with GET, POST, PUT, DELETE Requests.”
By implementation and deployment: Components can be implemented from the scratch or reused from any standard libraries such as .NET framework, JDBC, Java Swing Library etc. The components are usually integrated into one system through the interconnection between “require” and “provide” interfaces. As for software services they can be developed from the scratch or built based on the composition of other components. The Service provider publishes the Web Service at a discovery agency. The potential service requestor searches for a service at the discovery agency, acquires the URL of the required service, gets the WSDL file, builds the client and uses the provided service.
By the availability of standards: Components don’t usually follow any specific public standards. The services follow the predefined standards such as standard access protocol SOAP, standard methods of service description WSDL and subscription UDDI. All these standards provide consistent and compatible communication between different systems across the network.
By reusability: Components acting as basic blocks are used for integration of basic functionality whereas services being distributed over the network and having more powerful features can provide more functionality and extensibility after they are integrated into one system. Service-oriented applications can integrate separate business process and have more extensive functionality than component-oriented systems.
By methods of testing: Since most components are built locally they don’t have to be tested over the network. Standard white, black-box testing techniques can be used to test the components. As for services they have to be tested over the network and include performance, load, stress, usability, accessibility and security testing.

The major consequences of the listed above differences are the following:

  • The component-oriented systems can have less functional requirements, less maintenance than service-oriented systems
  • Since services can provide more functionality and can scale across the network (internet) the service-oriented systems can be more scalable than component-oriented systems.
