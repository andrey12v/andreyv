Why should conceptual modeling have multiple models, each expressing a different viewpoint of the overall system?

The conceptual model is an abstract view of the future system and cannot describe all aspects of the system implementation. Most systems consist of subsystems, components, packages and classes. Subsystems and components might have different relationships (aggregation, serialization, multiplicity etc.) and can be distributed differently across networks (local, wide etc.). The run time processes might interact differently during the execution of the application. All these aspects of the system are very important for consideration and should be described with various models. So the main reasons why the conceptual model should be supported by multiple models are the following:

  • The entire system cannot be expressed through one abstract view only because of its complexity. The system might consist of different subsystems, components and relationships on different levels of design. For example, UML Class Diagram can describe the structure of the system from its higher level representation (from components) to the lower level (individual classes). The Entity-Relationship Model represents database design. The distribution of the system components across the network is shown by means of the network model. The overall functionality of the system and its interaction with different stakeholders could be described by means of use case diagrams and business process models.
  • Multiple models, showing application from different viewpoints (business, architectural, technical), help to understand the system for different stakeholders. Multiple models improve communication between stakeholders and eliminate any ambiguities in requirements. For example, the model showing the functionality of the system on the business level is very helpful for business analysts and customers. The model expressing the technical design of the system is helpful for system analysts and development teams.
  • Multiple models help to resolve different possible issues of the future system and mitigate risks through prototyping, experimental testing. If the risk is hard to identify with one model then the scope of the area might be narrowed down and represented by means of other models. Multiple models help to view and predict functional and non-functional characteristics of the system (future performance, availability etc.)
  • Models expressing the system from different viewpoints help to manage the overall software development process more efficiently and implement set tasks on every stage of the software development cycle more accurately.   • The multiple models and their patterns could be reused in other software development cycles as example manuals or templates.
